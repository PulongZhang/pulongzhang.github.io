Fiona Fung 在 Anthropic 大会上讲了 28 分钟，聊了聊 [AI 时代到底该怎么管一个工程团队](https://www.youtube.com/watch?v=igO8iyca2_g)。其中有一个指标很值得单独拎出来看：**PR 所需的周期**。

这个指标看起来只是研发效率问题，实际上会同时暴露三类变化：团队对 AI 工具的接受度、代码产出速率的变化、以及下游工程基础设施是否还能承受新的提交频率。模型会继续变强，所以“哪些评审可以交给 AI、哪些必须由人审”的边界也不能一次性划死，而应该持续用数据重新评估。

## 问题：代码评审不能只看最终结果

我们现在想做的，是面向 PR 过程中的代码审批，建立一个更可靠的 Agent Workflow。它的核心目标不是“让 AI 随便看一遍代码”，而是让它的评审过程变成一条可审计的推理链：**先提出问题，再说明依据，接着接受验证，最后根据反证修正结论**。

代码评审和自动化测试不太一样。测试通常更接近一个是或否的问题：这条用例是否通过、这个接口返回是否符合预期、这段逻辑在给定输入下是否出错。但代码评审更像一个开放性问题。它没有唯一标准答案，也很少存在一组可以穷尽的检查项。一次评审有没有价值，不只取决于最后给出了几条评论，更取决于这些评论背后的推理逻辑是否成立：

- 是否理解了本次 PR 的业务意图；
- 是否找到了变更里的高风险路径；
- 是否解释清楚了问题为什么会发生；
- 是否区分了风格问题、真实缺陷和个人偏好；
- 是否能给出可复现、可定位、可修复的证据；
- 是否避免把无关问题、猜测问题当成阻塞项。

也就是说，代码评审里的 AI 不能只追求“看起来像一个评审结论”，而要能证明自己为什么这么判断。它更接近人类专家的工作方式：先理解上下文，再沿着风险路径提出假设，然后用独立视角验证这个假设是否成立，最后把站不住的评论剔除掉。

## 方法：从真实评审数据里训练 Workflow

当前比较合适的切入点，是以 [pr-review-toolkit](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/pr-review-toolkit) 为基底，再结合电能业务质量线的实际代码评审案例，做一个可度量、可迭代的评审工具。

这里最关键的不是先把工具做得多复杂，而是先拿到真实数据：

1. **PR 变更数据**：diff、相关文件、测试结果、CI 状态、依赖变更。
2. **人工评审数据**：人工 reviewer 提出的评论、最终采纳情况、修复提交。
3. **业务质量规则**：哪些问题在电能业务里是高优先级，哪些只是一般工程建议。
4. **团队风格约束**：命名、异常处理、日志、边界校验、测试习惯等公司内部偏好。
5. **结果反馈**：AI 找到的问题是否真实、是否重复、是否可操作、是否被开发采纳。

这些数据的作用，是把“AI 评审好不好”从主观感受变成可以对比的 benchmark：同一个 PR，人类发现了什么，Agent 发现了什么，哪些是真阳性，哪些是假阳性，哪些是人漏掉但后续被证明有价值的问题。

## Workflow 设计：发现问题之前，先设计检查过程

一个可用的代码评审 Agent Workflow，至少应该分成五个阶段。

### 1. 上下文收集

Agent 不能只看 diff。它需要先收集和本次变更相关的上下文，包括：

- PR 标题、描述和关联需求；
- 变更文件及其调用链；
- 相关测试、配置、数据库脚本、接口定义；
- 历史相似改动和最近缺陷；
- 团队已有的代码规范和评审偏好。

这一步的目标是判断“这次 PR 到底在改什么”，以及“哪些地方值得重点看”。如果上下文不足，后面的评审很容易变成泛泛而谈。

### 2. 多维度并行评审

单个 Agent 一次性完成所有检查，容易遗漏，也容易把关注点混在一起。更稳妥的方式，是把评审拆成多个维度并行展开，例如：

- 正确性：逻辑分支、边界条件、异常路径是否正确；
- 业务一致性：是否符合电能业务规则和现有产品语义；
- 可维护性：抽象是否过度、命名是否清晰、职责是否混乱；
- 测试充分性：是否覆盖关键路径、回归场景和失败分支；
- 性能与资源：是否引入不必要的查询、循环、锁或 IO；
- 安全与合规：是否涉及权限、敏感信息、越权访问；
- 团队风格：是否符合公司已有代码习惯。

每个维度都应该输出结构化发现，而不是一段泛泛的自然语言评论。

### 3. 证据化问题输出

每一条评审意见都应该至少包含四个要素：

- **位置**：文件、行号、相关函数或模块；
- **问题**：具体哪里不对，而不是“这里可能有问题”；
- **影响**：这个问题会导致什么后果；
- **修复建议**：建议如何改，是否需要测试补充。

如果一条评论无法定位、无法解释影响、无法给出验证方式，就不应该直接进入最终报告。代码评审最怕的是“看起来很聪明，但开发无法行动”的意见。

### 4. 优化机制

这个 Workflow 不应该只设计成“一个模型直接给出最终评审结论”，而应该拆成三个互相促进的部分：**验证器优化、生成器优化，以及两者协同**。

#### A. 验证器优化

验证器是整个系统的基石。它的任务不是生成新评论，而是判断一条评审意见是否成立、证据是否充分、严重程度是否合理。

但验证器一开始不能凭空优化，它首先需要一批人工标注的种子数据。比如从历史 PR 中抽样，把人工 reviewer 已经确认过的评论、被开发采纳的修复、以及明确被判定为误报的 AI 评论整理出来，由业务质量线或资深 reviewer 给每条评审意见打标签，并写清楚判断理由。

可以把验证结果分成三个等级：

- **1 分**：问题成立，证据充分，影响明确，应该进入最终报告；
- **0.5 分**：方向可能成立，但证据不足、影响不清，适合作为弱建议或交给人工确认；
- **0 分**：问题不成立、重复，或只是主观偏好。

这批人工标注数据的价值，不只是告诉验证器“这条评论对不对”，更重要的是告诉它“为什么对、为什么不对”。代码评审的判断标准往往藏在业务语义、历史约定和团队风格里，如果没有这批初始人工样本，验证器很容易把形式上像问题的评论当成真问题，或者把真正有风险但证据链较长的问题误判为误报。

但验证器本身也会犯错。它可能把真实问题误判为误报，也可能为了“显得严格”而过度挑刺。所以还需要一个 **Meta-Verifier**，专门审查“验证器的评价是否合理”。Meta-Verifier 不直接判断代码有没有问题，而是判断 verifier 的推理是否站得住：它有没有引用足够证据？有没有误读上下文？有没有把偏好当缺陷？

#### B. 生成器优化

生成器负责提出评审意见。它不仅要指出“哪里可能有问题”，还要同时输出自己的自我评价：这条评论的把握有多大，依据是什么，是否存在可能推翻它的反例。

这里的关键是鼓励模型诚实地暴露不确定性。如果模型发现自己的评审意见证据不足，它应该主动降级为“待确认”或撤回，而不是硬把它包装成确定缺陷。相反，如果它在证据不足时仍然强行下结论，就应该在训练和评估中被惩罚。

这样做的目的，是让生成器在提交最终评论前先完成一次自查：它不是盲目多报问题，而是尽可能把低质量评论过滤掉。

#### C. 生成器与验证器协同

随着生成器变强，它会提出更复杂、更隐蔽的问题，这反过来会提高验证器的优化难度。这里有一个关键点：验证器的能力必须始终略高于生成器，否则它就无法有效指出生成器评论中的漏洞。换句话说，生成能力提升以后，也要同步扩大验证侧的算力和样本难度，让验证器持续保持足够的批判能力。

这套机制后续还可以用于自动化标注。前期依赖人工标注建立验证器和 meta-verifier 的基本判断能力；当验证质量稳定后，再对新生成的评审意见进行多轮自动评估：让多个 verifier 独立判断，再由 meta-verifier 检查这些评估是否一致、理由是否成立。如果多数 verifier 都认为问题成立，并且 meta-verifier 也认可这些判断，这条评论就可以作为高质量样本；如果多数 verifier 认为它有缺陷，且理由一致，也可以反过来作为负样本。

在线上实际评审时，也可以引入两种优化策略。第一种是顺序修正：先生成一版评审意见，再让 verifier 找漏洞，生成器根据反馈修正评论，循环几轮后再输出最终结果。第二种是并行搜索：对高风险 PR 同时生成多组评审意见，再分别验证、打分、合并，优先保留证据最充分、影响最明确的评论。

这样，系统不是一开始就脱离人工专家，而是先用人工标注校准验证标准，再通过“生成评论—验证评论—审查验证—修正评论”的流水线，逐步扩大训练数据规模，也持续提高评审问题的难度和真实性。


### 5. 排序与最终报告

最终输出不应该是一长串平铺评论，而应该按风险和可行动性排序：

1. 阻塞合并的问题：真实缺陷、安全风险、数据错误、关键测试缺失；
2. 强建议修改的问题：明显可维护性问题、潜在性能风险；
3. 可选建议：风格一致性、命名、局部简化。

这样人类 reviewer 才能快速判断哪些必须处理，哪些可以后续优化。

## Benchmark：让 AI 评审和人工评审可比较

工具是否有效，不能只看“生成了多少条评论”。更应该看下面这些指标：

- **召回率**：人工高价值问题里，Agent 找到了多少；
- **准确率**：Agent 提出的问题里，多少被确认是真问题；
- **采纳率**：开发实际采纳或修复了多少；
- **重复率**：同一问题是否被多个维度重复报告；
- **可行动性**：评论是否能直接指导修改；
- **PR 周期变化**：是否缩短等待评审和返工时间；
- **基础设施压力**：提交频率提升后，CI 和环境是否成为新瓶颈。

特别是 PR 周期这个指标，需要拆开看。如果 AI 让代码提交变快，但 CI 排队、测试环境、人工审批变慢，总周期未必真的下降。那说明瓶颈已经从“写代码”转移到了“验证和集成”。

## 落地建议：先影子运行，再逐步接入流程

这个工具不适合一上来就作为强制门禁。更合理的路径是分阶段落地。

第一阶段，先做 **shadow mode**。Agent 在真实 PR 上运行，但不直接影响合并，只和人工评审结果做对比，积累 benchmark。

第二阶段，进入 **辅助评审模式**。Agent 可以在 PR 中输出建议，但默认由人类 reviewer 决定是否采纳。重点观察哪些维度最稳定，哪些维度误报最多。

第三阶段，再考虑 **分级门禁**。只有经过验证、准确率足够高的问题类型，才可以升级为必须处理的检查项。例如测试缺失、明显空指针、权限遗漏这类高确定性问题，可以比架构建议更早进入门禁。

同时，团队需要维护一套持续更新的评审知识库：

- 哪些问题类型已经证明高价值；
- 哪些误报模式需要抑制；
- 哪些公司风格规则应该固化；
- 哪些业务规则需要补充到评审上下文里；
- 哪些 Agent prompt 或 workflow 版本带来了效果提升。

## 结论

代码评审 Agent 的价值，不在于替代人类 reviewer，而在于把评审过程里可重复、可验证、可数据化的部分系统化。它应该帮助团队更早发现问题、更快聚焦风险，也让人类 reviewer 把注意力放到业务判断、架构权衡和复杂取舍上。

随着模型能力变化，这套 Workflow 也要持续迭代。真正重要的不是“AI 能不能评审代码”这个抽象问题，而是：在我们的业务、代码库、团队风格和工程基础设施条件下，哪些评审任务已经可以稳定交给 Agent，哪些还必须由人来把关。这个边界，需要用真实 PR 和真实数据不断校准。
